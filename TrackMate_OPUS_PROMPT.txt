FEATURE 1: REAL-TIME LOCATION TRACKING
────────────────────────────────────────────────────────────────────────────────

DESCRIPTION:
Every non-authority user (Tourist, Resident, Business) must continuously share
their GPS coordinates. The Authority dashboard must display ALL active users on
a live interactive map with colored role-based pins/markers.

SPECIFICATIONS:
- Location update interval: every 5 seconds (configurable)
- Map provider: Leaflet.js (OSM) or Mapbox GL JS
- Marker colors:
    Tourist  → Blue pin  (#3B82F6)
    Resident → Green pin (#10B981)
    Business → Orange pin (#F59E0B)
    Danger zone users → Red blinking pulse overlay
- Map must support zoom, pan, cluster markers when zoomed out
- Clicking a marker opens a sidebar panel with:
    • User name, role, Blockchain ID
    • Current coordinates (lat/lng)
    • Safety Score (live)
    • Current zone (safe/medium/high-risk)
    • Last seen timestamp
    • Button: "Send Alert to This User"
    • Button: "View Full Profile"
- Location data is streamed via WebSockets (Socket.IO)
- Each user's browser/device sends location via Geolocation API every 5s
- If GPS unavailable → use IP-based fallback (ip-api.com or similar)
- Authority sees a "Live Users" counter in the header (updates in real time)

OFFLINE HANDLING:
- If a tracked user goes offline (no ping for 30 seconds):
    • Their marker turns gray
    • Badge shows "OFFLINE"
    • Authority is notified in a toast notification
    • Last known location retained on map

IMPLEMENTATION STACK:
- Frontend: Socket.IO client (emit location every 5 seconds)
- Backend: Socket.IO server (broadcast to authority room)
- Storage: Redis (live location cache, TTL=60s) + PostgreSQL (location history)

FEATURE 3: ZONE MANAGEMENT SYSTEM (GEO-FENCING)
────────────────────────────────────────────────────────────────────────────────

DESCRIPTION:
Authority can draw, edit, and manage geographic zones on the map. Zones have
risk levels and automatically trigger alerts when users enter or exit them.

ZONE CREATION:
- Authority opens "Zone Manager" panel
- Draws polygon on map using Leaflet.Draw plugin
- Fills zone metadata form:
    • Zone Name (e.g., "Old City Market", "Industrial North")
    • Zone Type: Residential / Commercial / Tourist / Restricted / Industrial
    • Risk Level: LOW / MEDIUM / HIGH (with color coding)
        LOW    → Green fill  (#10B981, 20% opacity)
        MEDIUM → Yellow fill (#F59E0B, 20% opacity)
        HIGH   → Red fill    (#EF4444, 20% opacity)
    • Safety Index: numeric 0–100 (auto-calculated but editable)
    • Description / Notes
    • Active: toggle (disabled zones are grayed out)
- Save button → POST /api/zones
- Zone appears on map immediately for all users (broadcast via Socket.IO)

ZONE EDITING:
- Authority clicks existing zone → edit mode
- Can drag polygon vertices to resize
- Can change risk level, name, or deactivate zone
- PUT /api/zones/:id

ZONE DELETION:
- Soft delete (isDeleted flag)
- Removed from map, retained in audit history

GEO-FENCE TRIGGER ENGINE (BACKEND):
- Every time a user location is updated:
    1. Check which zones contain the new coordinates (point-in-polygon algorithm)
    2. Compare with user's previous zone set
    3. If user ENTERED a new zone:
        → Log zone_entry event
        → If zone risk ≥ HIGH → emit "zone_alert" to user's socket
        → Decrease safety score by 2 points per minute in HIGH zone
        → Notify authority: "User X entered HIGH-RISK zone: [Zone Name]"
    4. If user EXITED a zone:
        → Log zone_exit event
        → Recalculate dwell time
        → Update safety score accordingly

SAFETY INDEX CALCULATION (per zone):
    Safety Index = 100 - (incident_rate * 30) - (avg_severity * 20) - (high_risk_users_pct * 50)
    Range: 0–100, updated every 15 minutes via cron job

DESCRIPTION:
Each user has a Safety Score from 0 to 100 that changes dynamically based

SCORE REDUCTION RULES:
┌─────────────────────────────────────────────────────┬──────────────────────┐
│ Behavior                                            │ Score Change         │
├─────────────────────────────────────────────────────┼──────────────────────┤
│ Entering HIGH-RISK zone                             │ -2 per minute stayed │
│ Entering MEDIUM-RISK zone                           │ -0.5 per minute      │
│ Filing a CRITICAL severity incident                 │ -5 (one-time)        │
│ Being flagged by Authority                          │ -10                  │
│ Account suspended                                   │ Set to 0             │
│ Ignoring CRITICAL alert (not acknowledged in 5 min) │ -3                   │
└─────────────────────────────────────────────────────┴──────────────────────┘

SCORE RECOVERY RULES:
┌─────────────────────────────────────────────────────┬──────────────────────┐
│ Behavior                                            │ Score Change         │
├─────────────────────────────────────────────────────┼──────────────────────┤
│ Staying in LOW-RISK zone (per hour)                 │ +1                   │
│ Acknowledging CRITICAL alert promptly (< 2 min)     │ +2                   │
│ Incident resolved favorably (filed by user)         │ +3                   │
│ Account in good standing for 7 days                 │ +5 (weekly bonus)    │
│ Unflagged by Authority                              │ +5                   │
└─────────────────────────────────────────────────────┴──────────────────────┘

SCORE DISPLAY:
- Color coded gauge/ring on all dashboards:
    90–100 → Green  (Safe)
    70–89  → Teal   (Mostly Safe)
    50–69  → Yellow (Caution)
    30–49  → Orange (At Risk)
    0–29   → Red    (Danger)
- Animated score transitions (smooth number count-up/down)
- Score history chart (line graph, last 30 days) visible in user profile

CALCULATION ENGINE:
- Node.js worker process (or cron job every 60 seconds)
- Reads Redis for live zone dwell times
- Applies scoring rules
- Writes updated score to PostgreSQL users table
- Emits "score_update" via Socket.IO to user and authority

model User {

  email               String    @unique
  phone               String?

  nationalId          String?   // encrypted at rest
  emergencyContact    Json?     // { name: string, phone: string }
  safetyScore         Float     @default(100)
  status              UserStatus @default(ACTIVE) // ACTIVE | SUSPENDED | FLAGGED
  lastSeenAt          DateTime?
  lastLat             Float?
  lastLng             Float?
  currentZoneId       String?
  createdAt           DateTime  @default(now())
  updatedAt           DateTime  @updatedAt

  incidents           Incident[]
  alerts              UserAlert[]
  firs                FIR[]
  locationLogs        LocationLog[]
  zoneEvents          ZoneEvent[]
  safetyScoreLogs     SafetyScoreLog[]

6C. LOCATION BROADCAST LOOP (Server-side pseudocode)
────────────────────────────────────────────────────────────────────────────────

socket.on("location_update", async ({ lat, lng, accuracy }) => {
  const userId = socket.userId;  // set on auth middleware
  
  // 1. Cache in Redis (60s TTL)
  await redis.setex(`user:${userId}:location`, 60, JSON.stringify({ lat, lng, ts: Date.now() }));
  
  // 2. Log to PostgreSQL (every 5th update to reduce DB writes = every 25 seconds)
  if (socket.locationCounter % 5 === 0) {
    await prisma.locationLog.create({ data: { userId, lat, lng, accuracy } });
  }
  socket.locationCounter++;
  
  // 3. Zone detection
  const zones = await getActiveZones();  // cached in Redis, refreshed every 60s
  const currentZones = zones.filter(z => pointInPolygon([lat, lng], z.polygon));
  const prevZones = socket.currentZones || [];
  
  // 4. Detect entry/exit
  const entered = currentZones.filter(z => !prevZones.includes(z.id));
  const exited = prevZones.filter(id => !currentZones.map(z => z.id).includes(id));
  
  for (const zone of entered) {
    await handleZoneEntry(userId, zone);  // log event, adjust score, send alert
    io.to("authority").emit("user_entered_zone", { userId, zoneId: zone.id, riskLevel: zone.riskLevel });
  }
  for (const zoneId of exited) {
    await handleZoneExit(userId, zoneId);
    io.to("authority").emit("user_exited_zone", { userId, zoneId });
  }
  socket.currentZones = currentZones.map(z => z.id);
  
  // 5. Broadcast to authority
  const user = await getUserFromCache(userId);
  io.to("authority").emit("user_location", {
    userId, name: user.name, role: user.role,
    lat, lng, safetyScore: user.safetyScore,
    zoneId: currentZones[0]?.id || null
  });
});

SECTION 8: EDGE CASE HANDLING
================================================================================

────────────────────────────────────────────────────────────────────────────────
8A. OFFLINE USERS
────────────────────────────────────────────────────────────────────────────────
- If no location update for 30 seconds → mark user as OFFLINE in Redis
- Run a cron every 30s: scan Redis for stale location keys → emit "user_offline"
- Authority sees grayed-out marker with "OFFLINE" badge
- Queued alerts are delivered when user reconnects (read from UserAlert table)
- Safety score does NOT change while offline (no zone dwell calculation)

────────────────────────────────────────────────────────────────────────────────
8B. FAKE/SPOOFED LOCATION
────────────────────────────────────────────────────────────────────────────────
- Velocity check: if user teleports > 500km/h between updates → flag as suspicious
  → Authority notified: "Possible location spoofing: [User]"
  → Log event without updating displayed location
- GPS vs IP cross-check: if GPS city differs from IP geolocation city by > 100km
  → Flag for authority review (not automatic action)
- Accuracy threshold: if accuracy > 500m (low precision GPS) → show with accuracy ring
  → Authority sees low-accuracy badge on marker

────────────────────────────────────────────────────────────────────────────────
8C. HIGH LOAD SCENARIOS
────────────────────────────────────────────────────────────────────────────────
- Location updates: rate-limited to 1 per 5 seconds per user (Redis sliding window)
- Socket connections: Socket.IO with Redis adapter for horizontal scaling
- Location broadcast: authority receives batched updates (50ms debounce aggregation)
  Instead of emitting 1000 individual events, batch them into a single "bulk_location_update"
  payload every 100ms
- Database: Location logs written in batches (every 5th socket update)
- Zone polygon checks: cache all zone polygons in Redis as arrays, run in-memory
  (not a DB query per location update)
- API rate limiting: 100 req/min per user (express-rate-limit)
- Read replicas: Add PostgreSQL read replica for analytics queries (don't hit primary)

FINAL NOTES FOR OPUS

- Build this as a MONOREPO with /client (React) and /server (Node.js) folders
- Use TypeScript strictly (no any types except where unavoidable)
