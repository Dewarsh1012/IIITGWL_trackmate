1. WHAT YOU ARE BUILDING
You are building SafeTravel v3.0, a Unified Civic Safety Operating System. This is a complete ground-up rebuild that expands the original tourist-only safety monitor into a full civic infrastructure platform usable by every person who lives in, works in, or visits a geographic area. The rebrand is intentional and total — this is no longer a niche tourist tool. It is the safety layer for an entire community.
The platform serves four distinct user types simultaneously. Tourists are travellers passing through an area who need geo-fenced safety monitoring, digital ID verification, SOS triggering, and real-time itinerary guidance. Local residents are permanent inhabitants who need the ability to report incidents, track area safety scores, receive neighbourhood alerts, and access emergency services instantly. Business owners and vendors are shopkeepers, hotel operators, tour guides, and service providers who need to register their establishment, receive footfall and safety intelligence about their zone, and communicate safety updates to their customers. The product vision is a single platform that any person — regardless of whether they are passing through or born there — can open, register within 60 seconds, and immediately contribute to and benefit from a living safety mesh. Every alert that any user raises makes the platform smarter. Every zone boundary that an authority draws makes every user in that zone safer. Every incident report that a resident files adds a data point to the AI engine. The intelligence compounds across all four user types simultaneously.
This document is the complete specification. Build exactly what it describes. Do not make architectural shortcuts. Do not use mock data. Do not fake any integration. Every feature described here must be real, wired to the real database, and operational end to end.

2. ABSOLUTE RULES — READ BEFORE WRITING A SINGLE LINE
These are non-negotiable constraints. Every decision you make — every component, every query, every UI state — must be consistent with all of them simultaneously.
RULE ONE — NO MOCK DATA ANYWHERE: Every piece of data visible on any screen must originate from a real Supabase table via a real query. If no data exists in the database, the component renders an empty state with a helpful message and a call-to-action. A file named .env.example documents both variable names without values. The .gitignore excludes .env from version control. Any developer cloning the repository must supply their own .env file.
RULE THREE — ALL ROUTES ARE PROTECTED BY ROLE: The route /tourist/* requires the 'tourist' role. The route /resident/* requires the 'resident' role. The route /business/* requires the 'business' role. The route /authority/* requires the 'authority' role. An AuthGuard component enforces this for every protected route. Unauthenticated visitors are redirected to /auth. Authenticated users accessing a route that requires a different role see a clear Unauthorized screen, not a blank page or a runtime error.
RULE FOUR — NO SIMULATED MAPS: All maps are real Leaflet.js instances rendering live OpenStreetMap tiles. Geo-fenced zones are drawn as real polygon overlays sourced from the database. User positions are real GPS coordinates from navigator.geolocation. No CSS rectangles, no inline SVG fakes, no static image backgrounds passed off as maps.
RULE FIVE — ALL FORMS HAVE ZOD VALIDATION: Every input the user can submit has a Zod schema. Validation runs on change, not only on submit. Every field has an inline error message beneath it when invalid. No form submits to the database with invalid data under any circumstances.
RULE SIX — PII IS NEVER STORED RAW: Aadhaar numbers, passport numbers, and any other government-issued identifier are hashed with SHA-256 via the Web Crypto API before touching the network. Only the hash and the last four characters are stored in the database. The raw value is held only in memory for the duration of the registration flow and then discarded.
RULE SEVEN — REAL-TIME IS REAL: The authority dashboard and resident alert feed use Supabase Realtime channel subscriptions. New alerts appear without a page refresh. Tourist and resident position updates appear on the authority map without a page refresh. There is no polling. There is no manual refresh button for live data.
RULE EIGHT — ALL TEXT IS INTERNATIONALISED: Every string visible to the user is defined in a react-i18next translation bundle, never hardcoded in JSX. The platform ships with English, Hindi, Japanese, French, and Spanish bundles. A language switcher visible on every authenticated page allows the user to change language at runtime. The selected language is persisted to the user's profile in the database.

4. FOUR USER ROLES — DETAILED PERSONAS AND NEEDS
4.1 Tourist
Tourists are visitors to an area — domestic travellers, foreign tourists, pilgrims, adventure trekkers, and business travellers. They arrive at entry points like airports, bus stations, and checkposts. They may speak any of the five supported languages. They are unfamiliar with local geography and entirely dependent on the platform for safety guidance. Their primary needs are: a verified digital identity that authorities can validate without paper documents, a live map showing exactly which zones are safe and which are restricted, a panic SOS button they can trigger with one sustained press when in danger, step-by-step itinerary tracking so authorities always know where they are supposed to be, multilingual alerts when they are approaching a danger zone, and emergency contact notification when an SOS is triggered. Tourists are the most vulnerable user type and the platform must prioritise their safety above all other concerns.
4.2 Local Resident
Local residents are people who live permanently in the area. They know the geography intimately but face different safety challenges — neighbourhood crime, infrastructure hazards, crowd events, and community emergencies. Their primary needs are: the ability to file incident reports directly from their phone with GPS coordinates and optional photo evidence, a neighbourhood safety score that reflects the real-time incident density in their area, alerts when a nearby incident is reported or when a zone around them changes risk level, a community feed showing resolved and active incidents in their ward or locality, quick access to emergency services with one tap, and the ability to verify their residential status to receive ward-specific communications. Residents are the platform's most numerous user type and their collective reporting is the primary data source for the AI anomaly engine.
4.3 Business Owner and Vendor
Business owners and vendors are shopkeepers, hotel and guesthouse operators, tour guides, travel agents, restaurants, medical clinics, and any commercial entity serving both residents and tourists. Their primary needs are: a business registration flow that records their location, operating hours, services offered, and emergency contact; a dashboard showing footfall trends and safety conditions in their commercial zone; the ability to post safety advisories visible to tourists and residents in their vicinity; alerts when an incident is reported near their premises; integration with the tourist digital ID system so they can verify a tourist's identity and trip validity before providing services; and monthly safety reports for their zone that they can share with their staff. Business owners are the platform's most data-rich passive sensors — their foot traffic data combined with incident reports creates a dense picture of zone safety.
4.4 Authority
Authorities are government officers across multiple departments — district police, tourist police, disaster management, municipal ward officers, fire services, and ambulance dispatch. Multiple authority accounts can be active simultaneously, each scoped to their jurisdiction. Their primary needs are: a real-time god-view map of every active tourist, resident, and flagged business in their jurisdiction; an alert feed that aggregates SOS signals, AI-detected anomalies, resident incident reports, and zone breach events in real time ordered by severity; the ability to acknowledge, assign, escalate, and resolve any alert from any user type; tools to draw, edit, and delete geo-fenced zones with specific risk levels and automated rules; a tourist roster searchable by name, blockchain ID, nationality, and current location; the ability to generate an electronic First Information Report pre-populated from any alert or incident record; and analytics dashboards showing incident trends, zone performance, response times, and tourist flow patterns over time.

5. DATABASE SCHEMA — COMPLETE SPECIFICATION
Create all tables in a single migration file at supabase/migrations/001_initial_schema.sql. Enable Row Level Security on every table the moment it is created. Create all enum types before the tables that reference them.
Enable the pgcrypto and uuid-ossp extensions at the top of the migration file.
Define the following enum types before any tables. The user_role enum has values tourist, resident, business, authority, and admin. The id_type enum has values aadhaar, passport, voter_id, and driving_license. The alert_severity enum has values low, medium, high, and critical. The alert_status enum has values active, acknowledged, assigned, escalated, resolved, and false_alarm. The alert_source enum has values sos_panic, zone_breach, ai_anomaly, resident_report, authority_manual, and iot_band. The zone_risk_level enum has values safe, moderate, high, and restricted. The efir_status enum has values draft, submitted, under_review, resolved, and closed. The itinerary_stop_status enum has values upcoming, current, completed, and skipped. The business_category enum has values accommodation, food_beverage, transport, medical, retail, tour_operator, and other.
5.1 profiles Table
This table extends Supabase auth.users. The primary key id is a UUID that references auth.users(id) with ON DELETE CASCADE. The role column uses the user_role enum and has no default — it must be explicitly set at registration. The full_name column is TEXT NOT NULL. The phone column is TEXT. The avatar_url column is TEXT. The blockchain_id column is TEXT UNIQUE and stores the SHA-256-based cryptographic identifier. The id_type column uses the id_type enum. The id_number_hash column is TEXT storing the SHA-256 hash of the raw government ID. The id_last_four column is TEXT storing the last four characters for masked display. The safety_score column is INTEGER defaulting to 100 with a CHECK constraint ensuring it stays between 0 and 100 inclusive. The preferred_language column is TEXT defaulting to 'en'. The ward_id column is UUID referencing the wards table — used for residents and business owners. The is_verified column is BOOLEAN defaulting to false — flipped to true when an authority confirms the user's identity. The is_active column is BOOLEAN defaulting to true. The created_at and updated_at columns are TIMESTAMPTZ defaulting to NOW().
5.2 wards Table
Wards are administrative sub-divisions of the monitored area. Columns: id as UUID primary key, name as TEXT NOT NULL, district as TEXT NOT NULL, state as TEXT NOT NULL, center_lat and center_lng as DOUBLE PRECISION NOT NULL, geojson as JSONB storing the full polygon of the ward boundary, ward_officer_id as UUID referencing profiles(id) storing the assigned authority officer, and created_at. Wards are publicly readable. Only admins and authorities may insert or update ward records.
5.3 zones Table
Zones are geo-fenced safety areas that can overlap with wards and cross ward boundaries. Columns: id, name as TEXT NOT NULL, description as TEXT, risk_level using the zone_risk_level enum NOT NULL, center_lat and center_lng as DOUBLE PRECISION NOT NULL, radius_meters as DOUBLE PRECISION NOT NULL, geojson as JSONB for complex polygon shapes, managed_by as UUID referencing profiles(id) to identify the responsible authority, is_active as BOOLEAN defaulting to true, auto_alert as BOOLEAN defaulting to true controlling whether the AI engine generates alerts for this zone automatically, and created_at. Zones are publicly readable. Only authority and admin roles may insert or modify zone records.
5.4 location_logs Table
This is the highest-throughput table in the system. Every active user contributes a row every 30 seconds. Columns: id, user_id as UUID NOT NULL referencing profiles(id) with cascade delete, latitude and longitude as DOUBLE PRECISION NOT NULL, accuracy_meters as DOUBLE PRECISION, speed as DOUBLE PRECISION, heading as DOUBLE PRECISION, battery_level as INTEGER, is_in_safe_zone as BOOLEAN defaulting to true, current_zone_id as UUID referencing zones(id), current_ward_id as UUID referencing wards(id), source as TEXT defaulting to 'gps' with valid values of gps, iot_band, and manual, and recorded_at as TIMESTAMPTZ defaulting to NOW(). Create a composite index on user_id and recorded_at DESC immediately after the table creation. Create a second index on current_zone_id for zone-level queries. This table must also have a retention policy — rows older than 90 days should be pruned by a scheduled Edge Function to prevent unbounded growth.
5.5 incidents Table
Incidents are reports filed by residents, triggered by tourists, detected by the AI, or created manually by authorities. This is the central event log of the platform. Columns: id, reporter_id as UUID referencing profiles(id), incident_type as TEXT NOT NULL with documented values including crime, accident, medical_emergency, infrastructure_hazard, missing_person, suspicious_activity, crowd_emergency, natural_disaster, and fire; title as TEXT NOT NULL, description as TEXT, latitude and longitude as DOUBLE PRECISION, zone_id as UUID referencing zones(id), ward_id as UUID referencing wards(id), severity using the alert_severity enum NOT NULL, status using the alert_status enum defaulting to active, source using the alert_source enum NOT NULL, assigned_to as UUID referencing profiles(id) for the authority handling it, resolved_by as UUID referencing profiles(id), resolved_at as TIMESTAMPTZ, evidence_urls as TEXT array for uploaded photos or documents stored in Supabase Storage, metadata as JSONB defaulting to an empty object for supplementary data like AI confidence scores or IoT band readings, is_public as BOOLEAN defaulting to false controlling whether residents in the same ward can see this incident in their feed, and created_at.
5.6 trips Table
Trips apply only to tourists and define the validity window for a tourist's digital ID. Columns: id, tourist_id as UUID NOT NULL referencing profiles(id) with cascade delete, destination_region as TEXT NOT NULL, start_date as DATE NOT NULL, end_date as DATE NOT NULL, entry_point as TEXT, vehicle_details as TEXT, is_active as BOOLEAN defaulting to true, and created_at. A tourist's digital ID is considered valid when today's date falls within start_date and end_date and the trip's is_active flag is true.
5.7 itineraries and itinerary_stops Tables
The itineraries table has id, trip_id referencing trips with cascade delete, tourist_id referencing profiles, title as TEXT NOT NULL, and created_at. The itinerary_stops table has id, itinerary_id referencing itineraries with cascade delete, destination_name as TEXT NOT NULL, description as TEXT, latitude and longitude as DOUBLE PRECISION NOT NULL, planned_arrival and planned_departure as TIMESTAMPTZ, actual_arrival and actual_departure as TIMESTAMPTZ, status using the itinerary_stop_status enum defaulting to upcoming, safety_notes as TEXT, contact_name and contact_phone as TEXT for the point of contact at this stop, sort_order as INTEGER NOT NULL used for ordering, and created_at. Never order itinerary stops by id or created_at — always use sort_order.
5.8 businesses Table
Columns: id, owner_id as UUID NOT NULL referencing profiles(id) with cascade delete, business_name as TEXT NOT NULL, category using the business_category enum NOT NULL, description as TEXT, latitude and longitude as DOUBLE PRECISION NOT NULL, address as TEXT, zone_id as UUID referencing zones(id), ward_id as UUID referencing wards(id), operating_hours as JSONB storing a structured schedule by day of week, phone as TEXT, website as TEXT, is_verified as BOOLEAN defaulting to false flipped by authorities, is_active as BOOLEAN defaulting to true, logo_url as TEXT for a Supabase Storage link, and created_at.
5.9 emergency_contacts Table
Columns: id, user_id as UUID NOT NULL referencing profiles(id) with cascade delete, name as TEXT NOT NULL, phone as TEXT NOT NULL, relation as TEXT, is_primary as BOOLEAN defaulting to false, and created_at. Every user type can have emergency contacts. When an SOS alert is triggered, the platform sends an automated notification to all emergency contacts of the triggering user.
5.10 efirs Table
E-FIR stands for Electronic First Information Report. Columns: id, incident_id as UUID referencing incidents(id), user_id as UUID NOT NULL referencing profiles(id) — the subject of the report, filed_by as UUID NOT NULL referencing profiles(id) — the authority who created it, title as TEXT NOT NULL, description as TEXT NOT NULL, incident_type as TEXT NOT NULL, incident_location as TEXT, incident_lat and incident_lng as DOUBLE PRECISION, incident_time as TIMESTAMPTZ, status using efir_status defaulting to draft, evidence_urls as TEXT array, blockchain_hash as TEXT storing the SHA-256 hash of the complete FIR payload at the time of submission for tamper-proofing, witness_statements as JSONB storing an array of structured witness objects, and created_at and updated_at as TIMESTAMPTZ.
5.11 Row Level Security Policies
Enable RLS on all ten tables then define policies as follows. On profiles: a user may SELECT and UPDATE only their own row; authorities may SELECT all rows; admins have full access. On location_logs: a user may INSERT and SELECT only their own rows; authorities may SELECT all rows. On incidents: anyone may INSERT their own report; the reporter may SELECT their own incidents; authorities may SELECT all incidents and may UPDATE status, assignment, and resolution fields; residents may SELECT incidents in their own ward where is_public is true. On zones: SELECT is open to all including unauthenticated visitors; only authority and admin may INSERT, UPDATE, and DELETE. On wards: SELECT is open to all; only admin may INSERT or UPDATE. On trips: tourists may full CRUD their own records; authorities may SELECT all. On itineraries and itinerary_stops: tourists may full CRUD their own; authorities may SELECT all. On businesses: owners may full CRUD their own record; authorities may SELECT all and UPDATE the is_verified field; the public may SELECT verified and active businesses. On emergency_contacts: users may full CRUD their own contacts. On efirs: only authorities may INSERT and UPDATE; the subject tourist may SELECT their own e-FIR records.

6. AUTHENTICATION AND IDENTITY
6.1 Unified Auth Page
The /auth route renders a single page that handles registration and login for all four user types. The page has two tabs: Register and Login. The Register tab begins with a prominent role selection step — the user must choose Tourist, Resident, Business Owner, or Authority before seeing the registration form. Each role option is a large visual card with a representative icon, a role name, and a one-sentence description. Selecting a role reveals the role-specific form fields beneath. Authority registration requires an authority code — a short alphanumeric code that must match a code stored in a platform_config table to prevent unauthorised authority account creation.
6.2 Role-Specific Registration Fields
Tourist registration collects full name, ID type, ID number, phone, email, password, and confirm password. It also prompts for trip start and end dates and destination region, which creates the first trip record alongside the profile. Resident registration collects full name, phone, email, password, and confirm password, and requires the user to select their ward from a dropdown populated from the wards table. Business owner registration collects full name, business name, business category, address, phone, email, password, and confirm password, and requires selecting a ward. Authority registration collects full name, designation, department, phone, email, authority code, password, and confirm password.
6.3 Blockchain ID Generation
After every successful registration, generate a blockchain_id by constructing a payload object containing the Supabase user ID, the SHA-256 hash of the government ID number for tourists or the phone number for other roles, the user's role, the UTC timestamp of registration, and a platform salt string stored in an environment variable named VITE_BLOCKCHAIN_SALT. Serialise this object to a canonical JSON string, hash it with SHA-256 using the Web Crypto API, and take the first 16 characters of the hexadecimal representation, prefixed with 'BC' for tourist, 'RE' for resident, 'BZ' for business, and 'AU' for authority. Display this ID to the user in a styled confirmation card with a copy-to-clipboard button before any redirect. Persist it to profiles.blockchain_id.
6.4 AuthGuard Component
The AuthGuard component at src/components/AuthGuard.tsx accepts a required role prop typed as the user_role enum. On render it immediately calls supabase.auth.getSession(). If no session exists, it redirects to /auth with a redirectTo query parameter set to the current path so the user can be returned after login. If a session exists, it fetches the user's profile from the profiles table. While fetching it renders a full-screen centred spinner with an accessible aria-label. If the fetch fails or the profile does not exist, it signs the user out and redirects to /auth. If the profile role does not match the required role, it renders an Unauthorized screen that shows the user their actual role and provides a link to the correct dashboard. If the session and role both match, it renders children. Subscribe to supabase.auth.onAuthStateChange inside this component so sign-out events trigger an immediate redirect regardless of which page is active.
6.5 Session and Inactivity
Configure the Supabase client with persistSession true and autoRefreshToken true. Implement inactivity detection in a custom hook named useInactivityTimeout at src/hooks/use-inactivity-timeout.ts. The hook attaches event listeners for mousemove, keydown, click, and touchstart on the document. Each event resets a ref-stored timer. When the timer fires after 30 minutes of inactivity, it calls supabase.auth.signOut() and redirects to /auth?reason=inactivity. Show a warning toast 2 minutes before the timeout fires, giving the user a chance to click to stay signed in.

12. AI ANOMALY DETECTION ENGINE
Create a Supabase Edge Function at supabase/functions/detect-anomalies/index.ts. This function runs on a cron schedule every five minutes and is also invocable directly via HTTP for immediate triggering after an incident report is submitted.
The function performs six anomaly checks in sequence for every active user in the location_logs table who has contributed a location in the past hour.
Check one is Prolonged Inactivity: For every active tourist, if the difference between NOW() and their most recent location_logs.recorded_at exceeds 30 minutes and their last known status was not in a safe static location such as a registered accommodation, insert an incident row with source ai_anomaly, severity high, alert_type inactivity, and a description explaining the last known location and time gap.
Check two is Route Deviation: For every tourist with an active itinerary, compare their current GPS coordinates to the coordinates of their current itinerary_stop. If the Haversine distance exceeds 5 kilometres, insert an incident with severity medium and type route_deviation.
Check three is Unusual Speed: For every active user, compare the two most recent location_log rows. If the calculated speed between them using the Haversine formula divided by the time delta exceeds 130 kilometres per hour, insert an incident with severity high and type unusual_speed, as this may indicate a vehicle accident.
Check four is Zone Breach: For every active user, if their most recent location falls outside all active zones, insert an incident with severity medium and type zone_breach.
Check five is Resident Incident Cluster: If more than three incident reports are filed within a 500-metre radius within any 15-minute window, insert a synthetic incident of severity critical with type incident_cluster to alert authorities to a possible developing situation.
Check six is Safety Score Adjustment: After all checks, update profiles.safety_score for each tourist. Decrease the score by 10 for each active anomaly linked to them. Increase the score by 2 for each completed 30-minute window where they were consistently within a safe zone with no anomalies. The score is always clamped between 0 and 100.
Every anomaly-generated incident includes a metadata JSONB object containing the anomaly_type, the confidence score as a number between 0 and 1 reflecting the certainty of the detection, the specific data points that triggered it, and the timestamp of detection.

13. BLOCKCHAIN DIGITAL ID SYSTEM
The blockchain ID system is a client-side cryptographic identity mechanism that simulates the tamper-evidence of a distributed ledger without requiring an actual blockchain network. It works by making the ID deterministically derivable from the source data — any change to the underlying data changes the ID, and a mismatch proves tampering.
The implementation lives in src/lib/blockchain.ts and exports four functions.
The first function is generateBlockchainId which accepts an object containing userId, role, idHash, issuedAt as an ISO string, and a platform salt, serialises the object to a canonical JSON string with keys sorted alphabetically to ensure determinism, encodes the string as UTF-8 bytes using TextEncoder, passes the bytes through crypto.subtle.digest with the SHA-256 algorithm, converts the resulting ArrayBuffer to a hexadecimal string, takes the first 16 characters, and prepends a two-character role prefix. The function returns the final string.
The second function is verifyBlockchainId which accepts a DigitalId data object and an expected hash string, calls generateBlockchainId with the same data, and returns a boolean indicating whether the generated hash matches the expected hash exactly.
The third function is generateEFIRHash which accepts a complete e-FIR data object, serialises it to canonical sorted JSON, hashes it with SHA-256, and returns the full 64-character hexadecimal hash. This is stored in efirs.blockchain_hash on submission.
The fourth function is verifyEFIRHash which accepts the current e-FIR data object and the stored hash, recomputes the hash, and returns true if they match and false if any field has been altered since submission.
The public ID verification page at /verify/:blockchainId allows unauthenticated users to paste or type a blockchain ID, which triggers a Supabase query on profiles.blockchain_id. If a match is found, the page displays the holder's name, role, blockchain ID, issue date, and for tourists, the trip start and end dates and whether the ID is currently valid. The page displays nothing else — no contact details, no location history, no incident records.

14. REAL-TIME GPS TRACKING
Create a custom hook at src/hooks/use-location.ts that uses navigator.geolocation.watchPosition to subscribe to continuous GPS updates. The hook returns the current position object containing latitude, longitude, accuracy, speed, and heading, a loading boolean that is true until the first position fix is received, an error string if geolocation is unavailable or denied, and a manually-callable refresh function.
Every time a new position arrives from watchPosition, the hook inserts a new row into the location_logs table using the Supabase client. The row includes the current user's ID from the auth session, the latitude and longitude, the accuracy and speed and heading from the GeolocationCoordinates object, and the source field set to 'gps'. Before inserting, perform a client-side zone membership check by computing the Haversine distance from the new coordinates to each active zone's centre and comparing it to the zone radius. Set is_in_safe_zone to true if the point falls within any zone with risk_level of safe, and set current_zone_id to the ID of the matching zone with the lowest risk level. If the point falls outside all zones, set is_in_safe_zone to false and current_zone_id to null.
The hook debounces database inserts to a minimum of 30 seconds between writes regardless of how frequently the GPS updates. This prevents excessive database writes on high-accuracy GPS hardware without sacrificing the real-time map position which updates on every GPS event.
Handle the case where geolocation is denied by the user gracefully. Show a persistent banner on the tourist and resident dashboards explaining that location access is required for safety features and providing a button to open the browser's permission settings.
